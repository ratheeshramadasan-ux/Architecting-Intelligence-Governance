Agentic AI Governance — When Your Agent Does Something Wrong and Nobody Knows Who Is Responsible

A practitioner framework for AI use case qualification, model change governance, automated testing architecture, and portfolio capacity planning. Built from real enterprise failures in BFSI regulated environments.

⚖ Ratheesh Ramadasan 📅 June 2025 🕐 18 min read 🏦 BFSI · Enterprise Architecture · AI Risk

The Incident That Should Not Have Happened

A dental insurance company deployed an AI agent to process claims end-to-end. Testing was thorough. Accuracy was exceptional. The business case looked bulletproof.

The agent went live. For months — it performed flawlessly.

Then the model was upgraded.

A small sample was tested before the upgrade rolled out. The sample passed. The team moved forward. What nobody tested was the full scope.

Edge case claim types — rare in the test sample, common enough in production — started processing differently. Not wrongly enough to trigger alerts. Just wrongly enough to matter.

  • Claims that should have been escalated for human review were auto-approved
  • Claims that should have been paid were silently rejected
  • Patients were affected. Providers were unpaid
  • Compliance exposure followed

Thousands of transactions had been processed under unvalidated model behaviour before the pattern was caught.

The agent had not malfunctioned. The upgrade had not failed. The governance had.

This article dissects exactly what went wrong — and provides the complete governance framework to prevent it. Not as theoretical guidance. As operational controls with specific owners, evidence requirements, and classification.

Four Governance Failures in One Incident

The dental claim incident was not a technology failure. It was four governance failures occurring simultaneously, each of which, fixed independently, would have prevented the outcome.

Failure 1 — Wrong tool for part of the job

The frequency limit check — "procedure D0274 not payable within 12 months of prior D0274 claim" — is a deterministic rule. A single conditional statement. It should never have been inside the AI agent's decision scope. A rule engine handles it with zero governance overhead, zero model change risk, and 100% auditability. The AI agent should only have touched the ambiguous cases.

Failure 2 — Sample testing treated as full validation

A representative sample was tested before the model upgrade. Edge case claim types were absent from the sample. This is the most common and most dangerous governance shortcut in model change management. The vendor's description of a change as "minor" cannot be independently verified. Your test library is the only ground truth you control.

Failure 3 — No canary monitoring in production

Had known claim scenarios been running continuously against the live agent every 2–4 hours, the decision distribution shift would have been detected within hours of the upgrade — not weeks later. Volume does not compound when the alert fires on transaction 50, not transaction 50,000.

Failure 4 — No named owner of production governance

Go-live governance existed. Production governance — the ongoing monitoring, model change management, and quarterly review — did not. Nobody had asked: "What is our process when the model changes six months after go-live?" The question was not unanswerable. It was unasked.

Framework Part 1 — AI Use Case Qualification

The most expensive AI governance mistake is implementing AI for problems that do not require it. Every unnecessary AI deployment consumes governance bandwidth, increases regulatory exposure, and adds model change risk to a portfolio with finite capacity to manage it.

The Core Principle

Implement AI when it is required — not because it is easy to implement. For cognitive tasks requiring interpretation of ambiguous, unstructured, or multi-factor input, evaluate AI. For calculations that can be expressed as a formula, and processes that can be automated with rules or RPA — do not scope AI. The governance overhead must be justified by genuine capability need.

The Three Implementation Tiers

Tier 1 — Formula / Rule Engine
Deterministic. Calculable. No ambiguity.

The output is always the same for the same input. A formula produces it. A rule produces it. No judgement is required. This is never an AI use case. Examples: premium calculation, interest computation, SLA breach detection, frequency limit checks on procedure codes, duplicate record matching on exact fields.

Tier 2 — Automation / RPA
Structured. Repeatable. Process-driven.

The process steps are defined. The data is structured. The decision logic can be mapped. Automation executes it faster and more reliably than a human. AI adds governance cost with no capability advantage here. Examples: structured form data entry, file routing by type, scheduled report generation, straight-through payment processing with clean data.

Tier 3 — AI / Cognitive
Ambiguous. Contextual. Requires judgement.

The problem involves unstructured data, variable context, or decisions requiring interpretation across multiple factors simultaneously. A formula cannot produce the answer. Human judgement is currently required. Examples: free-text clinical note triage, multi-factor fraud behaviour analysis, ambiguous contract clause interpretation, complex multi-diagnosis claim assessment. Only Tier 3 problems justify AI deployment.

The AI Qualification Checklist — 8 Gate Questions

Every AI implementation request must pass all 8 questions before architecture approval is granted. The Review Board's default position is not approved. The project team proves AI is required — not the other way around.

# Gate Question If YES If NO
Q1 Can the output be produced by a deterministic formula or rule set for the majority of cases? Implement formula. Do not scope AI. Handle exceptions with human review. Proceed to Q2.
Q2 Is the input data structured and the process steps fully mappable? Implement RPA or workflow automation. AI adds governance cost with no capability advantage. Proceed to Q3.
Q3 Does the problem genuinely require cognitive interpretation — unstructured data, variable context, multi-factor judgement? AI may be appropriate. Continue qualification. Return to Q1 or Q2. Problem has been miscategorised.
Q4 Has a non-AI alternative been formally evaluated and documented as insufficient? Document why the alternative was rejected. This is the audit trail. Return to Q1. AI is not approved on the basis of convenience.
Q5 Is the organisation's current AI governance maturity sufficient to manage this deployment safely? Governance team confirms bandwidth available in writing. Queue. Do not deploy until maturity gap is closed.
Q6 Is this a high-risk domain — regulated decisions, financial transactions, patient impact, legal exposure? AI maturity requirement doubles. Require demonstrated production performance from a comparable deployment before approval. Standard qualification continues.
Q7 Can the AI decision be explained to a regulator, auditor, or affected individual in plain language? Document the explainability mechanism. Must exist at deployment — not promised for later. Blocked. Non-explainable AI in a regulated context is non-compliant before it goes live.
Q8 Does portfolio bandwidth capacity exist to govern, monitor, and respond to model changes? Governance team confirms in writing. Bandwidth score added to portfolio register. Queue. Portfolio at capacity. Not a negotiation.

Framework Part 2 — The Five Governance Control Categories

Category 1 — Model Sovereignty

If you cannot control, version-pin, or independently roll back the model your agent runs on — none of the downstream governance controls matter. A vendor who can push a silent update at midnight bypasses every control you have built.

G-01 Blocker
Version pinning

The agent must run on a pinned, named model version — never a floating alias like "latest". If the vendor updates the underlying model without changing the alias, your agent silently changes behaviour. Enforce at the API gateway level, not in documentation.

G-02 Blocker
Vendor change notification SLA

Contractual requirement for minimum 30-day advance notice of any model version deprecation or forced upgrade. Without this, a vendor retiring a model can force your agent onto an unvalidated version overnight.

G-03 Blocker
Tested rollback capability

The ability to revert the agent to the previous model version must be tested before go-live. If rollback takes more than 30 minutes or requires a deployment cycle, you do not have rollback capability.

G-04 Required
Open-weights preference for Tier 1

For Tier 1 agents processing regulated transactions, prefer open-weights models on controlled infrastructure. You own the weights, the version, and the upgrade decision. No vendor can push an unilateral update.

Category 2 — Model Change Management

The dental claim failure was a change management failure. The upgrade reached production with unvalidated edge cases. A full-scope re-validation gate would have caught it before deployment.

Why Class 1 and Class 2 both require 100% full library

The dental claim failure was a Class 2 change — a minor version increment. "Minor" is the vendor's assessment of what changed internally. You cannot independently verify it. Your test library is the only ground truth you control. Full scope validation is non-negotiable for any model version change regardless of vendor classification.

Category 3 — Automated Testing Architecture

Manual testing fails for agentic AI for one fundamental reason: agentic systems are probabilistic and operate at scale. An automated harness running 3,000 test cases in 45 minutes — with statistical drift detection across every domain — is not a convenience. It is the only way to govern non-deterministic behaviour at enterprise scale.

The test library must be structured in four tiers:

  • Tier A — Critical path: The 20% of transactions representing 80% of business risk. Runs in every test cycle. Results in under 30 minutes.
  • Tier B — Domain coverage: Every distinct transaction category — mapped to business domain taxonomy, not technical taxonomy. Minimum 5 cases per domain regardless of frequency.
  • Tier C — Edge cases and boundaries: Previously failed transactions, decision boundary cases, rare combinations. Never removed. Never deprioritised.
  • Tier D — Adversarial and regression: Every historical production incident becomes a permanent test case within 5 business days. The library only grows.
The Append-Only Principle

The test library is a versioned, append-only artefact. Deletion requires a formal decommission request with risk owner sign-off — not a developer decision. A test case that passes consistently is proof the control is working. It is never a candidate for removal. The dental claim edge cases that caused the failure are now permanently in Tier C.

The automated harness runs five layers simultaneously:

  1. Input replay engine — fires every test case against the candidate model via the same API gateway used in production. Runs each case 3–5 times to handle non-determinism.
  2. Output assertion engine — evaluates each response against expected output and tolerance thresholds. Flags all deviations without discarding ambiguous results.
  3. Statistical drift detector — compares decision distribution of the candidate model against the production baseline across every domain. A 3% drift on any domain triggers review — even if every individual test case passes.
  4. Regression comparator — runs the full Tier D suite. Zero tolerance: a single regression failure blocks the upgrade regardless of overall pass rate.
  5. Canary shadow runner — during parallel running, the candidate model processes a shadow copy of live production transactions. No candidate output reaches the business system. Discrepancies logged in real time.
Category 4 — Blast Radius and Containment
G-15 Blocker
Hard transaction caps

Maximum volume the agent processes in any rolling 4-hour window without human review. Enforced in integration layer code — not in the model prompt. A prompt instruction can be overridden. An integration layer cap cannot.

G-16 Blocker
Automatic suspension thresholds

If the agent's decision pattern on any transaction type deviates beyond a defined threshold in a 4-hour window — the agent suspends processing for that type and escalates automatically. Without requiring a human to notice first.

G-17 Blocker
Tested kill switch

Complete agent suspension achievable in under 5 minutes without a deployment cycle. The procedure is documented, activators named, and the switch tested quarterly. A kill switch that has never been tested is not a kill switch.

G-18 Required
Segregation of duties

The agent that processes the transaction cannot be the agent that marks it complete. A second validation layer — human or rule-based — sits between agent action and final commitment to the business system. Mirrors basic enterprise finance controls.

Category 5 — Accountability and Ownership
The Accountability Rule

A single named individual — not a team, not a function — is accountable for the agent's behaviour in production. They sign off on model changes, review monitoring reports, and are the first call when an incident is declared. "The AI team owns it" is not accountability. When the dental claim incident occurred, the absence of a named owner meant the question "who is responsible?" had no answer. That absence is itself a governance failure.

Framework Part 3 — Portfolio Capacity Governance

Every project team sees AI as a competitive requirement. Every roadmap now has an AI component. But the organisation's capacity to govern, validate, monitor, and respond to AI changes is finite — and nobody is managing it as a shared constraint.

The result: governance bandwidth is exhausted before the rollout calendar is half complete. Test cycles get shortened. Model changes get rubber-stamped. Windows for action compress. This is not a technology problem. It is a portfolio planning failure.

The Compressed Window Problem

Model change notifications arrive with 30 days notice — if you are fortunate. A Class 2 validation takes 7 business days. Parallel running takes 3 more. That is 10 days per agent. If six agents run on the same foundation model and the vendor upgrades simultaneously — which they do — six parallel validation cycles compete for the same governance team. At six projects, the 30-day window becomes effectively 5 days per project. That is not enough time to do it properly. This is the planning failure most organisations will not discover until they are already in it.

Model Diversification as Governance Strategy

Deliberately running different AI deployments on different foundation models is not just a vendor risk strategy — it is a bandwidth protection strategy. A vendor upgrade affects only agents pinned to that model. Diversification ensures upgrade cycles never stack. This is a governance argument for model diversification that most architecture decisions miss entirely.

Bandwidth Thresholds

  • Below 50% — Optimal. Governance controls properly executed. Model changes absorbed without shortcuts.
  • 50–70% — Strained. Everything works until something goes wrong. One incident consumes all remaining capacity.
  • Above 70% — Red line. No new AI deployments approved. Zero exceptions without CEO sign-off.
  • 20% buffer reserved at all times — never allocated to any project. The organisational equivalent of hospital emergency bed capacity.

The AI Review Board

Every AI implementation request passes through an independent Review Board — not the project team that proposed it. The project team has already decided they want AI. Their evaluation of whether AI is appropriate is not objective.

The Board's mandate is to challenge the qualification, not approve it. Six roles, none from the proposing project:

  • Risk function lead — challenges risk classification, verifies regulatory exposure assessment
  • Enterprise architect — evaluates whether the problem is genuinely cognitive or solvable by simpler means
  • CISO representative — reviews data security implications, prompt injection exposure, vendor contract adequacy
  • Legal / compliance — confirms regulatory compliance, reviews explainability obligations
  • AI CoE lead — confirms portfolio bandwidth availability, assigns governance team allocation
  • Independent business domain SME — evaluates whether the formula-first alternative was genuinely tested

Six Governance Principles

P1 — Implement AI when required, not because it is easy

The default answer to any AI implementation request is no. The project team proves AI is necessary — not the governance team proving it is not.

P2 — Formula first. RPA second. AI third.

If a deterministic formula or rule set solves the problem for 95%+ of cases, AI is not required. The governance overhead of AI must be justified by genuine cognitive capability need.

P3 — Every test case is permanent

The test library is append-only. Edge cases are never removed. A passing test case is proof the control works — not a candidate for deletion.

P4 — Go-live governance is not production governance

An agent that passed UAT is an agent that passed UAT. Production governance requires continuous monitoring, model change management, and quarterly review — indefinitely.

P5 — The vendor's "minor" is your governance risk

Any model version change triggers full re-validation regardless of vendor classification. You cannot verify what changed internally. Your test library is the only ground truth you control.

P6 — Bandwidth is a shared constraint, not a project resource

Governance team capacity for model changes, monitoring, and incident response is finite and managed at portfolio level. No project owns governance capacity. The portfolio does.

Closing

The dental claim incident was preventable at four separate points. Any one of those governance controls — the use case qualification gate, the full-scope re-validation requirement, the canary monitoring, or the named risk owner — would have changed the outcome.

What makes this framework different from published AI governance literature is not the individual controls. It is the recognition that governance bandwidth is finite, that AI is being scoped into more projects simultaneously than any organisation can govern properly, and that the most important governance decision is often the one made before the first line of agent code is written: does this problem actually require AI?

The answer, more often than the industry currently acknowledges, is no.

"The most expensive AI mistake is using AI because you can — not because you should."
